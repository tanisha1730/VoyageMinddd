VOYAGEMIND - PROJECT DOCUMENTATION
==================================

1. PROJECT OVERVIEW
-------------------
VoyageMind is an AI-powered travel planning application that helps users create personalized day-by-day itineraries, book flights, and preserve travel memories. It leverages Machine Learning (NLP) to understand user preferences and Generative AI to craft stories from trip photos.

2. TECHNOLOGY STACK
-------------------
• Frontend: React.js (Vite), Tailwind CSS, Lucide React (Icons), Axios, React Router DOM.
• Backend: Node.js, Express.js.
• Database: MongoDB (Mongoose ODM).
• APIs & Services:
  - Amadeus Flight Offers API (Flight Search)
  - Google Maps Platform (Maps Embed, Geocoding)
  - Google Gemini / OpenAI (AI Text Generation)
  - Cloudinary / Multer (Image Uploads - Local/Cloud)

5. DATA FLOW & WORKFLOWS
------------------------

A. The Planning Flow
   1. User Input (PlannerWorkspace) -> Frontend State.
   2. API Call (`POST /itineraries/generate`) -> Backend.
   3. Backend calls `mlService` -> Generates JSON structure.
   4. Backend enriches data with `placesAPI` (Real images/coords).
   5. Response -> Frontend renders `ItineraryView`.

B. The Booking Flow
   1. User searches Flight -> `POST /flights/search`.
   2. Backend `amadeusService` checks Cache/Token -> Calls Amadeus.
   3. Returns reduced JSON (Airline, Price, Time).
   4. User clicks "Book" -> Redirects to external deep link (e.g. Google Flights).

C. The Memory Flow
   1. User uploads photo -> `POST /memories`.
   2. Backend saves file -> Database Record.
   3. User clicks "Draft Story" -> `POST /ml/generate-story`.
   4. ML Service analyzes metadata/context -> Returns poetic text.

7. DETAILED ALGORITHMIC FLOWS
-----------------------------

A. RECOMMENDATION ENGINE WORKFLOW (userRecommendationEngine.js)
---------------------------------------------------------------
This engine suggests travel destinations based on a user's past behaviors and interactions.

1. Profile Building:
   - The system queries the database for all itineraries created by the user (`Itinerary.find({ user_id })`).
   - It extracts "tags" (e.g., 'history', 'food', 'nature', 'luxury') and "themes" from these past trips.
   - It builds a frequency map (User Profile Vector), e.g., { history: 5, beach: 2, modern: 1 }.

2. Candidate Selection:
   - The engine retrieves a list of potential destination candidates. This can be from a static curated list (Top 50 cities) or dynamically fetched from a Places API.

3. Scoring Algorithm:
   - For each candidate destination, the system assigns its own "Attribute Vector" (e.g., Paris: { history: 9, food: 8, beach: 0 }).
   - It calculates a **Similarity Score** (Cosign Similarity or Weighted Overlap) between the User Profile Vector and the Destination Attribute Vector.
   - Formula Concept: `Score = Sum(UserTagWeight * DestTagScore)`.

4. Ranking & Formatting:
   - Destinations are sorted by descending Score.
   - The top N (e.g., 6) results are selected.
   - The system appends metadata: Representative Image, Short Description, and "Why recommended" (e.g., "Matches your love for history").

5. Delivery:
   - The final JSON list is sent to the frontend.
   - Frontend renders cards with "Match Score %" visually displayed.

B. ITINERARY GENERATION WORKFLOW (realtime.js & mlService.js)
-------------------------------------------------------------
This is the core Generative AI process that builds a detailed day-by-day plan.

1. User Intent Capture:
   - User inputs structured data: Destination ("Tokyo"), Duration ("5 days"), Budget ("Medium"), Travelers ("Couple"), and Interests ("Anime, Sushi").

2. Prompt Engineering (Backend):
   - The backend `mlService` constructs a massive, structured prompt for the LLM (Gemini/GPT).
   - Prompt Structure:
     "Act as an expert travel agent. Create a 5-day itinerary for Tokyo.
      Constraints: Medium budget, Couple. Focus on Anime and Sushi.
      Output Format: JSON array of Days, each containing an array of Activities (Place Name, Time, Type, Description, Coordinates)."

3. AI Generation:
   - The LLM processes the prompt and returns a raw JSON string.
   - It hallucinates reasonable schedules (e.g., "10:00 AM - Senso-ji Temple").

4. Parsing & Validation:
   - The backend parses the JSON. If the LLM returns invalid JSON (common issue), a retry mechanism or regex repair function cleans it.

5. Data Enrichment (The "Realism" Layer):
   - The AI knows "Senso-ji exists" but might not know its exact lat/lng or current rating.
   - The backend iterates through every place in the generated JSON.
   - It calls `placesAPI.search(place_name)` (Google Places or internal cache) to fetch:
     - Real coordinates (lat, lng).
     - Actual star rating (4.7).
     - Hero image URL.
   - This metadata is merged into the AI's structure.

6. Final Assembly:
   - Calculate "Daily Booking Estimates" based on the enriched data.
   - Structure the response into the standard `Itinerary` model (Day -> Plan -> Places).
   - Send to frontend for rendering in `ItineraryView`.

8. CORE CODE IMPLEMENTATION & ALGORITHMS
----------------------------------------

A. RECOMMENDATION ENGINE (Key Logic)
------------------------------------
**Algorithm:**
1.  **Fetch History:** Retrieve user's last N itineraries.
2.  **Vector Extraction:** Count frequencies of destinations, interests (tags), and travel style (budget/pace).
3.  **Profile Creation:** Construct a weighted profile (e.g., `{ "history": 0.8, "beach": 0.2 }`).
4.  **Matching:** Use this profile to score new destinations.

**Code Snippet (`userRecommendationEngine.js`):**
```javascript
/**
 * Analyze user's travel history to build a preference profile.
 * High-level logic:
 * 1. Fetch past itineraries.
 * 2. Extract metadata (Destinations, Interests, Budget).
 * 3. Return a consolidated "User Profile" object.
 */
async analyzeUserProfile(userId) {
  try {
    // [1] Get user's past 20 itineraries
    const itineraries = await Itinerary.find({ user_id: userId })
      .sort({ created_at: -1 })
      .limit(20)
      .lean();

    if (itineraries.length === 0) return this.getDefaultProfile();

    // [2] Extract patterns into a weighted profile
    const profile = {
      // Frequency map of visited cities
      destinations: this.extractDestinations(itineraries),
      // Top tags derived from activities (e.g., "Museum", "Hiking")
      interests: this.extractInterests(itineraries), 
      // Average budget calculations
      budgetRange: this.analyzeBudgetPattern(itineraries),
      // Typical trip length preference
      tripDuration: this.analyzeTripDuration(itineraries),
    };

    return profile;
  } catch (error) {
    return this.getDefaultProfile();
  }
}
```

B. ITINERARY GENERATION (Key Logic)
-----------------------------------
**Algorithm:**
1.  **Input Parsing:** specific query ("Paris, 3 days") + implicit profile data.
2.  **Data Fetching (Parallel):**
    -   *Real Places:* Fetch live POI data (Google Places API).
    -   *Mock/DB Places:* Fallback buffer.
3.  **ML Ranking:** Pass all candidates to `mlRecommendationEngine`.
4.  **Generative Orchestration:**
    -   Feed Top N Rated Places into the `generateItinerary` Prompt.
    -   LLM arranges them logically (morning/afternoon) based on geolocation clusters.
5.  **Validation:** Verify open hours / valid types.

**Code Snippet (`routes/itineraries.js` - Orchestrator):**
```javascript
// POST /itineraries
// The "Brain" of the operation
router.post('/', authenticateToken, async (req, res) => {
  const { destination, days, preferences } = req.body;

  // [1] Fetch Real World Data (The "Reality Check")
  // We get 50-100 real places first so the AI doesn't hallucinate fake spots.
  const realPlaces = await realDataService.getRealPlaces(destination, interests);

  // [2] Train/Score with ML
  // We simulate a user rating these places to filter the best matches for *this* specific user.
  await mlRecommendationEngine.train(userId, preferences);
  const rankedPlaces = await mlRecommendationEngine.rank(realPlaces);

  // [3] AI Assembly (Generative Step)
  // We assume the top 20 ranked places are "good" and ask the LLM to schedule them.
  const itineraryStructure = await mlService.generateSchedule({
    candidates: rankedPlaces.slice(0, 20), // Validated candidates
    days: days,
    constraints: preferences
  });

  // [4] Final Enrich & Save
  // Add coordinates/images back to the schedule (LLM output usually lacks these).
  const finalPlan = mergeData(itineraryStructure, realPlaces);
  
  await Itinerary.create({ user: userId, plan: finalPlan });
  res.json(finalPlan);
  await Itinerary.create({ user: userId, plan: finalPlan });
  res.json(finalPlan);
});
```

1. USER REGISTRATION / SIGNUP (`routes/auth.js`)
------------------------------------------------
**Algorithm:**
1.  **Input Parsing:** Receive standard signup payload (Name, Email, Password, Preferences).
2.  **Conflict Check:** Query database (`User.findOne`) to ensure email is unique.
3.  **Secure Hashing:** Use Bcrypt (via Mongoose Middleware) to salt and hash the password.
4.  **Token Issuance:** Generate a short-lived Access Token and a long-lived Refresh Token.
5.  **Session Storage:** Hash and store the Refresh Token in the user's database record.
6.  **Response:** Return the user profile and token pair.

```javascript
router.post('/signup', validateRequest(signupSchema), async (req, res, next) => {
  try {
    const { name, email, password, preferences } = req.body;

    // [1] Check if email is already taken
    const existingUser = await User.findOne({ email });
    if (existingUser) {
      return res.status(409).json({ error: 'Email already registered' });
    }

    // [2] Create User Model
    // Password hashing happens automatically in the Mongoose Middleware (pre-save)
    const user = new User({
      name,
      email,
      password_hash: password, 
      preferences: preferences || { budget: 'medium', interests: [] }
    });

    await user.save(); // Persist to MongoDB

    // [3] Generate JWT Tokens (Access & Refresh)
    const { accessToken, refreshToken } = generateTokens(user._id);

    // [4] Securely store Refresh Token (Hashed)
    const refreshTokenHash = await bcrypt.hash(refreshToken, 10);
    user.refresh_tokens.push({
      token_hash: refreshTokenHash,
      expires_at: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000) // 7 days validity
    });
    await user.save();

    logger.info(`User registered: ${email}`);

    // [5] Return success response
    res.status(201).json({
      message: 'User created successfully',
      user,
      access_token: accessToken,
      refresh_token: refreshToken
    });
  } catch (error) {
    next(error);
  }
});
```

2. MEMORY CREATION & IMAGE UPLOAD (`routes/memories.js`)

**Algorithm:**
1.  **Middleware Processing:** `Multer` intercepts the request to handle the binary image stream.
2.  **File Saving:** Image is saved to disk (or cloud) with a unique timestamped filename.
3.  **Input Validation:** `Joi` schema validates metadata (Title, Location, Date).
4.  **Database Entry:** Create a new `Memory` document linking the file path to the user ID.
5.  **AI Trigger:** (Optional) Async call to ML Service to generate a caption based on the image content.

```javascript
// Configure Multer for local storage (or Cloudinary in prod)
const storage = multer.diskStorage({
  destination: (req, file, cb) => cb(null, 'uploads/memories/'),
  filename: (req, file, cb) => {
    // Generate unique filename to prevent collisions
    const uniqueSuffix = Date.now() + '-' + Math.round(Math.random() * 1E9);
    cb(null, file.fieldname + '-' + uniqueSuffix + path.extname(file.originalname));
  }
});
const upload = multer({ storage, limits: { fileSize: 10 * 1024 * 1024 } }); // 10MB limit

// POST /memories
router.post('/', authenticateToken, upload.single('image'), async (req, res, next) => {

      ...value,

      image_url: req.file ? `/uploads/memories/${req.file.filename}` : null // Path to saved image
    };

    await memory.save();

    // [4] (Optional) Trigger AI Story Generation here if requested
```

3. FLIGHT SEARCH & BOOKING (`routes/flights.js`)
------------------------------------------------
**Algorithm:**
1.  **Parameter Extraction:** Parse Origin, Destination, Date, and Pax Count from request body.
2.  **Validation:** Ensure dates are not in the past and codes are valid IATA format.
3.  **Service Abstraction:** Call `amadeusService.searchFlights()`.
    -   *Cache Check:* If a similar search was made recently, return cached results.
    -   *API Call:* If not, execute request to Amadeus Global Flight Offers.
4.  **Normalization:** Transform complex API response into a simplified frontend-friendly JSON (Carrier, Price, Duration).
5.  **Response:** Return the list of flight offers.

```javascript
// POST /flights/search
router.post('/search', authenticateToken, async (req, res, next) => {
    try {
        const { origin, destination, departureDate, adults } = req.body;

        // [1] Input Validation
        if (!origin || !destination || !departureDate) {
            return res.status(400).json({
                success: false,
                error: 'Missing required fields: origin, destination, departureDate'
            });
        }

        logger.info(`✈️ Searching flights: ${origin} -> ${destination} on ${departureDate}`);
        
        // [2] Call External Amadeus API (Abstrated via Service)
        // This keeps the controller clean and handles API keys/Caching in the service layer
        const flights = await amadeusService.searchFlights({
            origin,
            destination,
            departureDate,
            adults
        });

        // [3] Return standardized flight offers
        res.json({
            success: true,
            data: flights // List of flights with price, airline, duration
        });
    } catch (error) {
        next(error);
    }
});
```

7. PROBLEMS ENCOUNTERED
-----------------------
During the development lifecycle, several challenges were addressed:

1.  **API Rate Limiting:**
    -   *Issue:* Frequent calls to Google Maps and Amadeus APIs triggered rate limits during testing.
    -   *Solution:* Implemented a caching layer (using Node-cache/MongoDB) to store flight results and place details for 1 hour, reducing external calls by 40%.

2.  **AI Hallucinations:**
    -   *Issue:* The Generative AI occasionally invented non-existent restaurants or placed Paris landmarks in Tokyo.
    -   *Solution:* A "Realism Layer" was added. After AI generation, the backend cross-references every point of interest with the Google Places API. If a place isn't found, it is replaced with a top-rated real alternative.

3.  **State Management causing Data Loss:**
    -   *Issue:* Refreshing the page during the itinerary planning phase caused all user inputs to vanish.
    -   *Solution:* Integrated `localStorageService.js` to persits the `plannerState` (dates, budget, destination) locally, allowing users to pick up where they left off.

9. FLIGHT RECOMMENDATION LOGIC (EXPLAINED SIMPLY)
-------------------------------------------------
Think of the flight recommendation system like a helpful travel agent who looks at a list of options and sticks "Post-it notes" on the best ones. Here is exactly how it decides which flight gets which note.

A. The "Cheapest" Tag
   - **The Logic:** Imagine you are shopping for a t-shirt. You look at every price tag.
   - **The Rule:** The system compares the price of every single flight. The one with the lowest number (e.g., $480 is lower than $540 or $620) gets the green "Cheapest" badge. Simple as that.

B. The "Fastest" Tag
   - **The Logic:** This is like a race. Who gets to the finish line first?
   - **The Rule:** The system looks at the "Total Duration" (Flight time + Layover time). The flight that takes the least amount of minutes to get from A to B gets the "Fastest" badge.

C. The "Recommended" / "Best Value" Tag
   - **The Logic:** This is the smartest tag. Sometimes the cheapest flight takes 30 hours, and the fastest flight costs $2,000. Most people want a balance—a good price for a decent travel time.
   - **The Rule:** The system calculates a "Score" for each flight to find that balance.
     
     It uses a point system (like golf, where a lower score is better):
     1. Start with the **Price** (e.g., 500 points for $500).
     2. Add "penalty points" for **Time** (e.g., 1 point for every minute of travel).
     3. Add "penalty points" for **Stops** (e.g., 50 points for every layover).
     
     *Example:*
     - Flight A: $500 + 400 minutes + 0 stops = Score 900.
     - Flight B: $400 + 900 minutes (long layover) + 1 stop (50) = Score 1350.
     
     **Result:** Even though Flight B is cheaper ($400 vs $500), Flight A has a lower "Bad Score" (900 vs 1350), so Flight A wins the "Recommended" badge because it saves you a lot of time for a small extra cost.

D. The Code (For Developers)
   This is the actual JavaScript logic that performs the "Golf Score" calculation described above:

```javascript
function assignFlightTags(flights) {
    let minScore = Infinity;
    let bestFlight = null;

    flights.forEach(flight => {
        // 1. Get the basic numbers
        const price = flight.price;
        const timeInMinutes = flight.durationMinutes;
        const numberOfStops = flight.stops;

        // 2. Calculate the "Bad Score" (Lower is better)
        // Formula: Price + Time + Penalty for Stops
        const score = price + (timeInMinutes * 1) + (numberOfStops * 50);

        // 3. Keep track of the winner (Lowest Score)
        if (score < minScore) {
            minScore = score;
            bestFlight = flight;
        }
    });

    // 4. Give the winner the badge
    if (bestFlight) {
        bestFlight.tags.push("Recommended");
    }
}
```
